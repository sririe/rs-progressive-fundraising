## Fulfillment Workflow and Process

### End-to-end digital order flow (invoice-driven)
- Receive digital order as invoice (e.g., CP114-4055; invoice #55; maté group)
- Download invoice to Downloads; run Python script to parse invoice and generate order folders
- Output: per-merchant folders (Amazon, Loblaws, Esso) with timestamped subfolders
- Shared storage: Lloyd's Google Drive; access for Doug and Mario

### Inventory sourcing vs generation
- Esso: use pre-purchased digital inventory (copy provided URLs into Excel; no generation)
- Amazon: generate card PDFs and package in zip; or produce URL list via SystemOne
- Loblaws: mixed history; older stock required field cleanup; moving to vendor-provided URLs
- Walmart: generated by Progressive (process to be shown by Mario)

### Invoice-to-Excel prep (Esso example)
- Open generated Excel in Esso folder; locate required denominations (e.g., two $100s, one $250)
- Copy from inventory files: invoice number, customer, activation date, URLs
- Alternate row colors for readability; update "removed from inventory" markers
- Save and add encryption password (customer mnemonic; no letters/dashes per current convention)
- Validate password by reopening before sending

### Amazon generation flow
- Prepare inventory Excel: denominations needed (e.g., 25s, 50s, 100s, 250s; one $10)
- Run Amazon generator script (Python) from desktop
- Verify outputs:
  - Per-denomination folders contain expected counts (e.g., 5 x $100, 2 x $25, 2 x $50, 2 x $250, 1 x $10)
  - Gift card PDFs look correct
  - GiftCardLog.xlsx created inside zip (catalog used by SystemOne)
- SystemOne step (for URLs):
  - Upload generated zip; reject if duplicate filename
  - Export report; receive Excel with one URL per card
  - Add encryption password; validate by reopening

### Customer delivery formats
- Two options for generated cards: encrypted zip or URL Excel
- For vendor-provided inventory (e.g., Esso), deliver as provided (usually URLs)
- Deliver one encrypted Excel/zip per merchant; example order had 4 separate files

## Scripts, Tools, and Environment

### Tooling and dependencies
- Python scripts; open-source libraries; Inkscape for PDF rendering
- No Git or formal version control; versions stored on Drive; changes logged in Google Calendar

### Operator experience and errors
- Mario runs same scripts; initial issues due to Python version differences on his machine
- Known behavior: console outputs, retries (e.g., Loblaws site slowness), and occasional parsing variance from invoice format differences
- Validation practices:
  - On-screen counters during generation and zip creation
  - Final folder count checks per denomination
  - Encrypted file reopen test

### Stability and brittleness
- Manual select-and-edit steps are accurate due to operator vigilance; low error incidence
- Risks increase with scale or multiple operators
- Proposed improvement: standardize inventory file schema to enable automation of data pulls and reduce manual copy/paste errors

## Inventory Management and Replenishment

### Process and triggers
- Just-in-time approach; trigger to fulfill is payment received
- Replenishment decision owner: Doug; Mario now also performs
- Daily lookbacks: compare to same period last year; adjust by 2 weeks to 1 month outlook by category
  - Grocery: never a month out due to high velocity
- For large unanticipated orders: procure before fulfillment workflow starts

### Inventory staging (Mario vs Lloyd folders)
- Merchant delivers new inventory to a staging area ("Mario's folder")
- When needed for orders, Mario moves required denominations into the merchant's live inventory
- Counter columns in inventory sheets track remaining quantities (e.g., 38 -> 33 after pulls)

## File Handling, Formatting, and Encryption

### Excel handling standards
- Freeze top row for scrolling clarity
- Per-merchant per-denomination Excel files
- Alternating row color convention to spot ordered items quickly

### Encryption practices
- Add password to each deliverable (Excel/zip)
- Password pattern: consistent per client; described to customer; validated before sending

## SystemOne and URL Publishing

### Purpose and flow
- Used to convert generated card PDFs into hosted URLs
- Inputs: zip containing PDFs and GiftCardLog.xlsx
- Steps: upload zip (unique name), then export URLs; deliver encrypted URL Excel to customer

### Operational notes
- If customer prefers zips, skip SystemOne and send encrypted zip
- Only applicable to generated-card flows (e.g., Amazon); vendor-inventory flows use provided format
